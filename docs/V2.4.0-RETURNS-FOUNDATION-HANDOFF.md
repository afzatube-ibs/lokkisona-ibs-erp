# v2.4.0 Returns Foundation — Handoff

**Release:** `2.4.0` · **Label:** Returns Foundation  
**Date:** 2026-06-11  
**Prior:** `2.3.0` Daily Dispatch Supplier Statement Foundation

---

## Purpose

Returns are **independent** from Daily Dispatch. Supplier Return Statement is a **return adjustment snapshot** — not a ledger document.

| Document | Accounting meaning (v2.5+) |
|----------|------------------------------|
| Daily Dispatch Statement | Increases supplier payable |
| **Supplier Return Statement** | Decreases supplier payable (Return Adjustment / Less Returns) |
| Supplier Payment | Decreases supplier payable |

**Net Payable** = Dispatch Total − Return Adjustments − Payments

No ledger posting in v2.4.0.

---

## Flow

1. **Staging** — Order at `hub_return` / `order_returning` → Pending Supplier Confirmation on `/return-receive`
2. **Supplier confirmation** — `confirmReceive` → `ibs_return_receives.status = received` (no payable impact)
3. **Ready for report** — Hub + Customer returns on `/return-reports` checkbox list
4. **Create Return Report** — one-step lock → `ibs_return_reports` + items; receives → `reported`
5. **Print** — `/return-report/{ref}/print` — manual Print click only

Lokkisona warehouse returns: confirm on Return Receive only; **excluded** from Return Report.

## Guards

- Duplicate return in report — UNIQUE `return_receive_id` + server check
- Missing reason — required at confirm; stored on receive when column exists
- Missing cost/rate — blocked at report create
- Mixed supplier / business source — blocked server + client
- Max 50 returns per report

---

## Legacy

- **Return Batch** UI removed from `/return-receive` (v0.7.x grouped batches remain in DB)
- Supplier Payables “Return Batch Deductions” card marked legacy; use Return Reports for new work
- `confirmReceive` no longer creates per-order batch/batch_item rows
