---

## Accounting chain (supplier payable)

End-to-end supplier balance is built in this order:

```text
Dispatch  →  Return  →  Payment  →  Balance
```

| Step | Document | Effect on supplier payable |
|------|----------|----------------------------|
| 1 | **Daily Dispatch Statement** | Increases payable (locked dispatch cost snapshot) |
| 2 | **Supplier Return Statement** | Decreases payable (**Return Amount** — locked return snapshot) |
| 3 | **Supplier Payment** | Decreases payable (cash/settlement) |
| 4 | **Balance Summary / Ledger** | Running net position |

**Net Payable** = Dispatch Total − Return Amounts − Payments

v2.4.x: Return Reports are snapshot-only — no `return_deduction` ledger rows yet.

## Locked cost snapshot (v2.5 ledger hook)

When v2.5 posts `return_deduction` to Supplier Ledger:

- **Use:** `ibs_return_report_items.product_cost_snapshot` and `ibs_return_reports.total_adjustment_amount`
- **Use:** order-line `supplier_cost_snapshot` / order `cost_snapshot_total` captured at dispatch/confirm time
- **Do not use:** current `ibs_products` supplier cost or live catalog price

At report create, `ReturnReportWriteService` stores `DispatchCostSnapshot::forOrder()` into each report item. Detail/print views reconcile displayed line totals to that locked amount even if catalog costs change later.

## Guards (one supplier + one business source)

| Layer | Enforcement |
|-------|-------------|
| Server | `ReturnReportWriteService` — “All selected returns must belong to the same supplier / business source.” |
| Client | `app.js` — disables submit when multiple `data-supplier` or `data-business-source` values are selected |

Verify with `tools/verify-return-report-guards.php` (dev helper).
