**Release tag:** `1.2.1`

**Date:** 2026-06-10

Read-only metrics hardening — dashboard KPIs no longer cap at 50. **No UI, routes, schema, or business-logic changes.**

---

## Problem fixed

`DashboardReadService` used `count($repo->findByStatus($status, 50))`, which capped every count at **50** because `findByStatus` applies `LIMIT 50`.

Affected surfaces:

- Owner dashboard: Active Orders, Shipped Awaiting Dispatch, Order Status Breakdown, Needs Attention
- Supplier dashboard: all workflow task cards, Pending Return Receive

---

## Solution

New methods on [`OrderWriteRepository`](app/Repositories/Write/OrderWriteRepository.php):

| Method | SQL |
|--------|-----|
| `countByStatus(string $status)` | `COUNT(*) WHERE ibs_status = :status` |
| `countByStatuses(array $statuses)` | `COUNT(*) WHERE ibs_status IN (...)` |
| `countReturnPending(string $returnType, string $ibsStatus)` | Same exclude logic as `findReturnPending`, without LIMIT |

[`DashboardReadService`](app/Services/ReadOnly/DashboardReadService.php) now uses these methods. Active fulfillment statuses are defined once in `ACTIVE_FULFILLMENT_STATUSES`.
